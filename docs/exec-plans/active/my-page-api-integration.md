# My 페이지 API 연동

## 목표

- My 페이지의 dummy 기반 설정/프로필 흐름을 실제 API 기준으로 전환한다.
- 프로필 수정, 설정 저장, 계정 삭제 흐름을 정리한다.

## 범위

- `src/pages/My.tsx`
- 관련 profile / settings API 호출부

## 현재 상태

- 아바타 업로드/삭제는 이미 연결되어 있다.
- 닉네임 저장은 `useUpdateMyProfile`로 연결되어 있다.
- 타이머 설정 저장과 자동 이월 토글은 `useUpdateMySettings`로 연결되어 있다.
- 프로필/설정 초기값은 `useGetMyProfile`, `useGetMySettings`로 받아온다.
- 계정 삭제는 `useDeleteMe`로 연결되어 있다.
- 페이지 내부에 프로필/설정 UI와 업로드 로직이 함께 모여 있다.

## 작업 단계

1. 계정 삭제 API 연동과 삭제 이후 후처리 흐름을 확정한다.
2. 삭제 이후 세션 정리와 라우팅 책임 위치를 정한다.
3. 필요하면 profile / settings 섹션을 분리한다.
4. 쿼리 invalidate / store sync 흐름을 정리한다.
5. 완료 후 auth/settings 문서를 갱신한다.

## 이번 업데이트 반영 메모

- `My` 페이지는 이제 프로필/설정 조회를 서버 데이터 기준으로 초기화한다.
- 닉네임 저장 성공 시 프로필 쿼리를 invalidate 한다.
- 설정 저장과 `auto_carry_todo` 토글도 서버 mutation 기준으로 동작한다.
- 아바타 업로드/삭제 성공 시 auth store와 프로필 쿼리를 함께 갱신한다.
- 회원 탈퇴 성공 시 auth store를 로그아웃 상태로 정리하고 랜딩 페이지로 이동한다.

## 주요 결정 포인트

- 계정 삭제 후 라우팅과 세션 정리를 어디서 담당할지
- My 페이지 내부의 query/store 동기화 로직을 페이지에 둘지 훅으로 분리할지

## 리스크

- 아바타 업로드 흐름과 일반 저장 흐름이 뒤섞이면 페이지 책임이 다시 커질 수 있다.
- 삭제 이후 세션 정리와 라우팅이 auth 흐름과 엮인다.

## 남은 작업

- 페이지 책임 분리 여부 판단

## 완료 조건

- My 페이지 주요 액션이 실제 API와 연결된다.
- 계정 삭제 이후 흐름이 명확히 정리된다.
- query/store 동기화 규칙이 문서화된다.
